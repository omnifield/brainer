# Brief — 🕳 fix: resume после рестарта — залипший «running» + жадный спавн на каждую сессию

| | |
|---|---|
| **Scope** | `backend` (`packages/backend/`) |
| **Owner** | owner-backend (запускает user, можно через пульт) |
| **Класс** | дыро-фикс (найдено догфудом пульта 2026-07-12: user видит «working» у сессии, завершившейся вчера) |

## Факты (диагностика architect)

1. **Статус переживает рестарт как ложь.** `hub._project`: `done` → `waiting`,
   активность → `running`. Если backend умер до прилёта `done` (или вместе с
   turn'ом), в sqlite остаётся `running`; `resume_all` на старте восстанавливает
   его КАК ЕСТЬ — но никакой turn рестарт не переживает (SDK-resume = новое
   подключение). Итог: вечный «running» без хода; UI показывает «working»
   (наблюдалось: `backend-5cb58f77`, updated_at вчера, «running» сегодня).
2. **Жадный resume = утечка процессов.** `resume_all` спавнит SDK-клиент
   (claude-процесс) на КАЖДУЮ персистентную не-stopped сессию при КАЖДОМ старте
   backend'а. Реестр только растёт → каждый рестарт плодит больше холостых
   процессов (наблюдалось: 2 живых claude по ~40 мин без дела).
3. **Зомби.** `[claude] <defunct>` в контейнере — дочерний процесс не пожат
   (гигиена disconnect/stop у адаптера/SDK).

## Fix (направление)

1. **Reconcile при resume:** персистентный `running` при восстановлении
   проецировать в `waiting` (turn не переживает рестарт — факт), `stopped`
   не резюмить вовсе. Честного «running» после рестарта не существует.
2. **Lazy resume:** клиент поднимать не на старте backend'а, а при первом
   обращении к сессии (subscribe / send / stop). `resume_all` вырождается в
   «загрузить реестр в память» (проекции/handle), без спавна процессов.
   Blueprint В2 требовал переживаемость рестарта, НЕ жадный прогрев — сверься
   с `briefs/control-channel-kernel.md`, если найдёшь противоречие — STOP
   и эскалация, не решай контракт сам.
3. **Гигиена процессов:** при `done`/`stop`/shutdown убеждаться, что дочерний
   claude завершён и пожат (нет `<defunct>`); shutdown-хук уже есть — проверить
   полноту.

## Verify (DoD)

- Тест: store с сессией `running` → resume → статус `waiting`; `stopped` не
  резюмится.
- Тест/факт: после старта backend'а с N персистентными сессиями — НОЛЬ
  claude-процессов до первого обращения; обращение поднимает ровно одну.
- Живой прогон: рестарт uvicorn при завершённой сессии → в UI «waiting», не
  «working»; `ps` в контейнере без `<defunct>` после stop.
