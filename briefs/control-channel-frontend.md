## Цель

Чат с живой агент-сессией в control-panel: стрим событий + отправка сообщений.
Это ядро догфуд-пересадки («мы сами пересаживаемся с терминалов на brainer» —
§0 direction-брифа): если в этом чате работать неудобно — продукт не готов.

## Deliverables

1. **Типы событий — ТОЛЬКО генерацией** (З3 ревью, канон types-from-schema):
   TS-типы событий/handle генерятся build-шагом из JSON-схемы kernel-пакета.
   Ноль ручных доменных типов; инструмент генерации — на твой выбор, требование —
   один источник (схема). Разъезд схемы и типов должен ломать build, не прод.
2. **Чат-вью сессии**:
   - подписка на SSE-стрим с reconnect (Last-Event-ID = `seq`; после reconnect —
     без потерь и без дублей);
   - рендер по типам: `message` — реплики диалога; `thinking` — свёрнутый блок
     (раскрываемый); `tool-call` + `tool-result` — парой по `call_id` (компактно:
     имя инструмента + статус, детали по клику); `status`/`done` — состояние сессии;
     `error`/`limit` — заметно, не тостом-однодневкой;
   - send-бокс: отправка в живую сессию, состояние «агент работает» между send
     и ответом.
3. **Интеграция в существующий дашборд**: чат — вью выбранной сессии (список
   сессий уже есть из interface-MVP), не отдельное приложение. Launch-флоу
   расширяется параметрами role/model по контракту ручек backend'а.
4. **Доки** — README: поток данных (SSE → генерированные типы → вью), генерация типов.

## Миграция на фреймворк (HCA) — писать сразу с расчётом

Фронт brainer'а позже пересаживается на наш фреймворк (канон
`commons/standards/canon/`: layers HCA, component-model, kit-first, tokens).
Kit (`@omnifield/*`) сюда сейчас НЕ тянем — но новый код кладём так, чтобы переезд
был **пере-домовкой файлов, не переписыванием**:

1. **Логика ленты — чистый модуль без UI** («UI is a Shadow»). Редьюсер событий
   (лента по 9 типам, seq-дедуп, reconnect-склейка, пары tool-call/result по
   `call_id`, статус «агент работает») — plain TS в `store/`, тестируется без DOM.
   Это будущие Controller/Feature. Solid-реактивность — тонкой обёрткой сверху,
   не вплетена в редьюсер.
2. **Каждый тип события — свой презентационный компонент, props-only**
   (будущие View/Shape): `components/chat/` — реплика, thinking-блок (свёрнутый),
   tool-пара, status/done, error/limit, заглушка permission-request. Экран
   (`screens/`) их только композирует (будущий Widget/Page) — ноль вёрстки
   сущностей в экране, ноль инлайн-рендера событий.
3. **Сгенерированные из JSON-схемы типы = единственный доменный источник**
   (deliverable 1) — это заготовка Entity-слоя. Никаких параллельных ручных
   интерфейсов «для удобства вью»: вью ест контрактные типы as-is.
4. **Прото-токены вместо raw-style**: цвета/spacing/radius чата — css-переменные
   в одном месте (`index.css`), компоненты ссылаются на переменные; ноль
   инлайн-стилей и магических значений в JSX. Состояние (loading/active) — класс
   из JS, не ручной style.
5. **API/side-effects только в `api/`** (SSE-клиент, send) — как уже сделано
   в interface-MVP; компоненты и store к fetch/EventSource не прикасаются.

Смок-тест миграции: замена chat-компонента на kit-пресет должна быть заменой
импорта + маппингом props — если для этого придётся распутывать логику из JSX,
раскладка неверна.

## Вне скоупа

Комнаты/группы/история (это chater, шаги 2–3 очереди — текущий чат живёт на стриме
сессии без своей персистенции переписки); ростер по пресетам (параллельный трек,
отдельный бриф); терминальный путь и существующие вью — НЕ выпиливать.

## Verify (DoD)

- Реальный e2e: открыть сессию → видеть живой стрим (реплики, свёрнутый thinking,
  пары tool-call/result) → отправить сообщение → увидеть ответ в том же треде →
  обрыв соединения/refresh → лента восстановилась без потерь и дублей.
- Юнит: редьюсер ленты по типам событий (все 9 типов словаря, включая
  `permission-request` — рендер-заглушка «зарезервировано»), reconnect-дедуп по `seq`.
- Build падает при рассинхроне с JSON-схемой (проверить намеренной поломкой в тесте CI).
- Тесты + lint зелёные; доки.
- Раскладка проходит секцию «Миграция на фреймворк»: редьюсер без DOM-зависимостей,
  события рендерятся именованными props-only компонентами, ноль инлайн-стилей
  (ревью-критерий architect).

## Заметки

- Git: commit-only, `feat(frontend): ...`; push/merge — architect после ревью.
- Контракт ручек/событий не устраивает → STOP + architect, не подгоняй типы руками
  и не пиши локальные обходы поверх генерированных типов.
