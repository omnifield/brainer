| **Порядок** | ПАРАЛЛЕЛЬНЫЙ трек к очереди control-канала (по решению direction-брифа §7); в зоне backend — после `control-channel-backend.md` (WIP=1 внутри зоны) |

## Цель

Роль-модель = данные: декларативный пресет в корне управляемого репо
(`.omnifield/preset.yaml`), поставочные пресеты копией при init, детерминированная
валидация, ручки для UI (диагностики + ростер). Хардкод зон перестаёт быть носителем
модели.

## Deliverables

1. **Схема пресета v1** — blueprint §2.2 (roles/rules/flow) + **запиненная семантика
   `rules` (З2, обязательна)**:
   - разрешение матча: most-specific-path wins; при равной специфичности — порядок
     списка; `match: "."` — fallback-правило корня, НЕ «всё дерево»;
   - при `per_match: true` scope-id инстанса = смэтченный путь относительно корня
     (напр. `packages/chater`); он же значение `OMNIFIELD_SCOPE` и ключ
     `{match}`-подстановки в persona;
   - инвариант: роль `main` обязательна.
   - Схема — JSON-схемой как артефакт (единый источник для валидатора и будущей
     генерации типов, тот же принцип, что у событий канала — З3).
2. **Валидатор** — детерминированные диагностики, без LLM: `unmatched-dir`,
   `dead-rule`, `schema-error` (blueprint §2.3) + **`overlap-rule`** (З2: два правила
   претендуют на одну папку при равной специфичности). Каждая диагностика — с путём
   /правилом-виновником, пригодна для показа в UI as-is.
3. **Поставочные пресеты** — `presets/` в brainer: `default-3arch` (продукт/фронт/бэк
   + owner'ы), `solo` (main + owner'ы), `flat` (без бэк/фронт-разреза). Материализация
   **копией** в `.omnifield/preset.yaml` управляемого репо (команда/ручка init);
   синхронизация копий — вне скоупа (уедет с харнессом §6).
4. **Ручки для UI**: валидация репо↔пресет (список диагностик) + ростер-резолв
   (скан папок репо → матч против rules → список доступных агентов: роль, scope-id,
   model по умолчанию). Контракт ручек делят фронт+бэк — менять только через architect.
5. **Стыковка с launch**: launch control-канала берёт роль из пресета репо
   (permission-уровень, model, persona) вместо захардкоженных зон.
6. **Доки** — README: схема, семантика матчинга (З2 дословно), диагностики.

## Вне скоупа

Адаптация хуков харнесса (`.claude/hooks/*` как читатели пресета) — architect по
синку §6 с оракулом, НЕ трогать; UI ростера/выбора пресета — отдельный фронт-бриф;
enforcement `flow`-рёбер (в v1 — декларация); синхронизация материализованных копий.

## Verify (DoD)

- Юнит на семантику матчинга: специфичность, tie-break порядком, `.`-fallback,
  `per_match` scope-id — по одному тесту на каждое правило З2, плюс все четыре
  диагностики валидатора (позитив/негатив).
- Реальный e2e: init пресета в тестовый репо → ростер-ручка отдаёт агентов по его
  папкам → намеренная рассинхронизация папок → диагностики её ловят.
- Launch с ролью из пресета проходит против живого канала (permission-уровень и
  persona реально применяются).
- Тесты + lint зелёные; трейсы на скане/резолве; доки.

## Заметки

- Git: commit-only, `feat(backend): ...`; push/merge — architect после ревью.
- Захотелось поля в схеме, которого нет в blueprint §2.2 → STOP + architect
  (у схемы три читателя: хуки, UI, валидатор — расширение только через контракт).
