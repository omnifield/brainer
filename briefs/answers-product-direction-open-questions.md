## 0. ГЛАВНОЕ (упущено в direction-брифе, исправляем): ядро brainer ≠ Claude Code

Brainer — **не продукт для работы с Claude Code**. Ядро ориентировано на **нашу экосистему**
(self-hosted agent-loop поверх `backend/llm` — ADR 065 ф.4-5 / 074). **Claude API/SDK — это
ИНТЕГРАЦИЯ**, ровно того же ранга, что GitHub-борды у tasker.

Принцип экосистемы (user, P0): *все наши продукты самодостаточны и зависят только от нашей
экосистемы; любые сторонние API — уровень интеграций.*

Следствия для твоего blueprint'а:

- **Контракт control-канала = НАШ, живёт в kernel** (провайдер-агностик): словарь событий
  (message / tool-call / tool-result / status / done / error / limit / permission-request),
  операции (launch / send / stream / resume / stop) — проектируются как будущий контракт
  НАШЕГО agent-loop. Интеграции в него мапятся, не наоборот.
- **Зеркало tasker-правила:** SDK/Claude-понятия наружу адаптера НЕ выпускать. Твой
  surface-пункт «SDK vs stream-JSON» переквалифицируется: `claude-agent-sdk` — ок, но как
  **деталь реализации claude-code-АДАПТЕРА**; его типы/семантика не пересекают границу
  адаптера и не формуют контракт. «Сверить shape событий SDK» = проверка реализуемости
  маппинга, не источник словаря.
- **Прагматика MVP не меняется:** единственная живая реализация сейчас = claude-code
  интеграция (наш agent-loop ещё не построен). Но ядро (kernel/orchestrator/UI) про Claude
  не знает. Тест на чистоту: появление self-hosted провайдера не должно тронуть ничего,
  кроме регистрации нового адаптера.

## В1. Permissions — решено: (a) + резерв в контракте

- **MVP = (a):** permission-mode задаётся **ролью пресета** на старте сессии (встаёт рядом с
  правами git в роль-модели §2). Никого не спрашивает.
- **(b) форвардинг в чат — горизонт** (записать, не строить).
- **НО:** событие `permission-request` **зарезервировать в словаре контракта уже сейчас** —
  наши self-hosted агенты тоже будут спрашивать; claude-code адаптер в MVP его просто не
  эмиттит (резолвит режимом). Дешёвая строка в словаре сейчас против ломки контракта потом.

## В2. Живучесть — решено: (b), это НЕ раздув

Persist session-реестра + resume после рестарта backend'а — принято; без этого догфуд-пересадка
(§0 direction-брифа) не случится, ты прав. Уточнение по границе: **resume — операция НАШЕГО
контракта**; как её исполняет провайдер (SDK resume у claude-code / свой state у self-hosted) —
дело адаптера. Форма persist'а (файл/sqlite) — твой выбор, минимализм ок.

## По surface-пунктам — ок, с поправками на §0

- OTEL-развилка (activity/status со стрима канала, OTEL = метрики) — ок, один источник правды.
- Материализация пресета **копией** при init — ок, ровно принцип самодостаточности.
- Лимиты v0 реактивно (фейловер по limit-ошибке) — ок; limit-ошибка при этом = **typed-событие
  контракта** (см. словарь §0), чтобы router-фейловер был провайдер-агностик.
- Терминальный путь не выпиливать до шагов 1–3 — ок, впиши в owner-брифы.
