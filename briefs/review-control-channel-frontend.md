## Ф1 (баг, обязателен): hard-refresh на `/sessions/:id` отдаёт голый 404-JSON бэка

Vite-прокси `"/sessions" → :8000` заглатывает и **HTML-навигацию** SPA-роута
`/sessions/:id`: FastAPI отвечает `{"detail":"Not Found"}`, приложение умирает.
DoD-пункт «refresh → лента восстановилась» не выполняется буквально.

Корень: коллизия неймспейса SPA-роута и API-поверхности за одним origin. Контракт ручек
(§1.5) НЕ меняем. Фикс на стороне фронта, механизм на выбор owner: `bypass` в vite-proxy
(проксировать только не-HTML Accept) или увести SPA-роут с `/sessions/*` (например
`/s/:id`). Тот же выбор обязан работать в prod-раскладке за одним origin — задокументируй.

## Ф2 (контракт, ратифицировано): user-реплика обязана жить в wire, не только оптимистично

Сейчас echo юзера — только optimistic-локально; после reconnect/replay реплика юзера
**исчезает из ленты** (провайдер её не эмиттит). Словарь контракта уже имеет
`message {role: user}` — резерв используем:

- **backend** (мелкий fixup): адаптер на `send()` эмиттит `message {role: user, text}`
  в стрим сессии (seq — его же, порядок «user-реплика до ответа» гарантирован местом
  инжекта). Попадает в ring-буфер → replay честный.
- **frontend**: при приходе wire `message {role:user}` вытеснять соответствующий
  optimistic-элемент (механизм дедупа — на owner), рендер уже готов (role из payload).

## Ф3 (минор): статус в шапке сессии не обновляется по `done`

Список (проекция бэка) после `done` показывает Waiting, шапка чата — Running до
следующего `status`-события. Редьюсер: `done` → `sessionState = reason === "stopped"
? "stopped" : "waiting"` — зеркало серверной проекции.
