## Сверка (для контекста, не вопросы)

Контракт в брифе = фактический kernel 1-в-1 (9 событий `events.py`, операции
`contract.py`, permission-словарь, sqlite-реестр с opaque `provider_state`).
Плюс находка: **fake-провайдер уже существует** — `kernel/tests/test_fake_provider.py`
(implementability-proof на чистом контракте). Деливерабл «fake» = поднять его из
теста в скриптуемый провайдер со сценариями, не с нуля.

## Вопросы / предложения

### В1. Размещение слоя B — осознанное отступление от канона kernel (блокирует blueprint)

Канон kernel README: «адаптеры живут у потребителя шва (backend)». Но продукт до
стаба не трогаем. Предлагаю: слой B (native-провайдер) — инкубационная зона
`packages/` рядом с движком, extract-candidate, ноль импортов из продукт-пакетов;
в README kernel — строка об осознанном отступлении (не дрейф). Подтверди/поправь.

### В2. Имена и нарезка зон (блокирует git-gate/scope wiring)

Слой A приходит от тебя пакетом — имя зоны? (`packages/llm-engine`?) Слой B —
отдельная зона (`packages/agent-loop`?) или подпакет kernel? Мой дефолт: две
отдельные зоны, у каждой свой owner-бриф и место в `claude-scope.ps1` + CLAUDE.md.
Скажи, как назвал пакет в порте, — подстроюсь.

### В3. Песочница прогона = devbox-контейнер (стык с вашим docker-флоу)

Тулсет v0 (read/write/exec) должен быть замкнут. Предлагаю: прогоны агентов над
chater — **внутри devbox-контейнера, путь clone-in-volume** (2б): хост недосягаем
классом, machine=cattle, заодно закрывается хвост devbox-теста. Следствия для
вашего с user разговора по докеру:
- агенту в контейнере нужны git-креды на chater (gh auth путь из фидбека) и
  claude-креды НЕ нужны (native-провайдер, инференс локальный);
- полигон ходит в слой A за `/llm/generate` — см. В4 (куда ходить);
- из devbox-фидбека на эту тему: pwsh в образе нет (Д9 — если лаунчер прогона
  останется ps1), forwardPorts у CLI-пути не публикует порты.
Полный фидбек: `devopser/briefs/feedback-devbox-brainer.md`.

### В4. GPU-раскладка слоя A

3070 Ti. Движок — хост-процесс (как voice/image в капсуле сейчас?) или контейнер
с CUDA (WSL2)? От этого: URL движка для полигона (`localhost` vs
`host.docker.internal` — второе уже проверено, работает) и чей это деплой-заказ
(devopser?). Мне всё равно куда ходить — нужен только зафиксированный адрес/env.

### В5. Conformance-suite: где гоняются небесплатные прогоны

fake — CI, ок. local — CI-раннер без GPU его не потянет: локальный таргет по
флагу/nx-таргет вне CI? claude-code (замороженный, платный) — только ручной
смок по требованию? Предлагаю: suite один, три профиля запуска
(`fake`=CI-гейт, `local`+`claude`=локальные nx-таргеты, в CI не ходят).

### В6. chater: права агентов

Кто выдаёт агентам git-права на `omnifield/chater` (PAT/gh с какими скоупами),
и действует ли там наш газ-гейт паттерн (owner-роли, commit-only) с первого дня —
или полигон стартует свободнее и ужесточаем по мере стаба?

## Не вопросы (заявляю, возражай если что)

- В fake закладываю сценарии `permission-request` — событие зарезервировано
  контрактом, но живьём нигде не эмитится; suite заставит его работать.
- resume native-провайдера — свой state в `provider_state` (opaque JSON),
  реестр готов, kernel не трогается.
- Blueprint слоя B (цикл, тулсет v0, схема tool-call) покажу вам двоим до
  owner-брифов, как требует координация исходного брифа.
