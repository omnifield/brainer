## Зачем этот бриф

Каркас MVP есть (interface-MVP + backend-MVP смержены, PR #1/#2). Дальше продукт растёт
по пакету решений ниже. Твоя работа: развернуть это в тех-контракты и порезать на
owner-брифы в заданном порядке. Fix-брифы в полёте (repo-registry, port 3500) — доделываются
до/параллельно, они мелкие.

## 0. Принцип-рамка: догфуд = продуктовый флоу

**Наш флоу == флоу юзера.** Ничего «у нас работает, у юзера нет». Следствия:
- brainer для репо — **опция**: жёсткой зависимости от brainer у управляемых репо нет
  (харнесс вендорится и удаляется, репо живёт без него);
- юзер может жить вообще без нейронок: только `claude-code` со своим акком — или без brainer;
- как только появляется чат (см. §1) — **мы сами пересаживаемся с терминалов на brainer**.
  Если нам неудобно — продукт не готов.

## 1. Control-канал — переворот orchestrator'а (⚠️ amends `backend-mvp.md`)

В backend-mvp было «инъекция сообщений в живую сессию — НЕ MVP». Это **пересмотрено**:
чат — обязательный функционал, а чат требует канала управления. Решение:

- Сессии спавнятся **headless** (Claude Agent SDK / `claude -p` stream-JSON режим), НЕ
  терминальный claude-scope. Терминал перестаёт быть местом жизни агента.
- Orchestrator = **спавнер + двусторонний канал** (send prompt → stream ответа/tool-активности),
  а не читатель телеметрии. **OTEL остаётся только для observability** (метрики/статус), не
  для управления.
- Существующий `IAgentProvider` расширяется каналом: `send(session, message)` +
  подписка на поток событий сессии. Shape канала — твоя проработка (свериться с тем, что
  реально отдаёт SDK/stream-JSON: assistant-сообщения, tool-use события, result).
- claude-scope.ps1 остаётся как ручной fallback и как референс env (OTEL-инъекция, scope).

Это самый первый шаг очереди: без канала chater — пустые комнаты.

## 2. Флоу-пресеты: роль-модель = данные, хуки = интерпретаторы

Сейчас роль-модель зашита в хуки (`scope-resolve.mjs` с хардкод-зонами). Переворачиваем:

- **Пресет** = декларативный конфиг **в корне управляемого репо** (версионируется с репо,
  команда шарит). Содержит:
  - **роли**: имя, модель по умолчанию, права git (full / commit-only / none), KB-доступ,
    system-prompt (≈ персона);
  - **правила «папка → агент»**: напр. `frontend/`+`backend/` → 2 архитектора,
    `packages/*` → owner на пакет;
  - **флоу-рёбра**: кто кому брифует, где ревью-гейт.
- **Main-архитектор есть ВСЕГДА** (инвариант любого пресета).
- Наша модель «3 архитектора (продукт/фронт/бэк) + овнеры-исполнители» = **дефолт-пресет
  в поставке**. Рядом обязательно мыслим solo-пресет (один main + овнеры) и пресет без
  чёткого бэк/фронт-разреза — юзер волен работать по-другому, не везде есть бэк и фронт.
- Хуки харнесса (scope-resolve/scope-identity/git-gate) становятся **читателями пресета**,
  не носителями модели.
- **Фронт идёт от пресета**: юзер задаёт рут репо → выбирает пресет → UI сканит папки,
  матчит против правил → показывает ростер доступных агентов. Запуск агента — с выбором
  модели.
- Схема пресета (формат, имя файла, валидация) — твоя проработка. Валидация
  «папки ↔ пресет расходятся» = детерминированная проверка, показывается в UI.

## 3. Chater — отдельный продукт, инкубируется здесь. Язык = **Go** (решено)

**Что это:** generic-мессенджер (горизонт — телеграм-подобный: комнаты, ГРУППЫ, история),
**своя БД**, про агентов НЕ знает. Brainer делает мост: агент-сессия = **participant** чата
(комната на сессию; группа = user + несколько агентов — напр. архитектор + овнер в одном треде).
История переписки живёт в chater, brainer своё хранение транскриптов не изобретает.

**MVP-минимум (анти-раздув):** комнаты + сообщения + история + participant-API для моста.
БЕЗ вложений, реакций, поиска, тредов — телеграм-паритет = горизонт, не сейчас.

**Инкубация (extract-candidate, паттерн kernel):** `packages/chater/` —
- ноль импортов из brainer-внутренностей (orchestrator/backend); допустимы только
  kernel-контракты через интерфейс;
- своя БД + свои миграции + свой конфиг;
- brainer-backend потребляет ТОЛЬКО по API — как будто chater уже в другом репо;
- пометка extract-candidate в README пакета + фенс-зона в scope-resolve.

**Go:** новый тулчейн в репо — CI-job, пины, lint/format (golangci-lint или что выберешь —
твоя проработка). ⚠️ КБ-канона Go ещё нет — эскалируй оракул-архитектору потребность в
статьях (структура сервиса/миграции/стиль) ДО первого owner-брифа на Go-код, чтобы owner
не работал без канона.

## 4. Tasker — фасад сейчас, полноценный продукт в перспективе

- Brainer говорит **только с tasker-контрактом**. Контракт — **tasker-native**
  (task: id/title/status/assignee/brief-link; операции list/get/move — минимум, расширишь
  по нужде UI). **GitHub-понятия (issue-номера, Projects-поля, draft-карточки) наружу не
  выпускать** — живут только в адаптере и мапятся в контракт, не наоборот.
- **v0 = тонкий прокси над GitHub Projects** (org `omnifield`, board «Migration» projects/1) —
  своё писать сейчас незачем, git-борды покрывают. GitHub = **одна из интеграций**, не основа.
- `packages/tasker/` — тот же фенс-паттерн что chater (изоляция, API-only, extract-candidate).
  БД у v0 нет (прокси). Стек v0 = имеющийся (py/ts) — новый язык в репо один за раз (Go у chater).
- ⚠️ GitHub GraphQL (Projects API) флапает на батчах — ретраить по одному (грабля проверена).

## 5. Мульти-акк = первый живой кейс router'а (kernel просыпается)

У user 2 акка Claude (личный + продуктовый), хочет кидать личные ресурсы на продукт и
работать несколькими сразу. Решение:

- **Аккаунт = ресурс.** Провайдер `claude-code` получает **инстансы по акк-профилям**:
  изоляция через `CLAUDE_CONFIG_DIR` per сессия (каждый профиль залогинен один раз).
- **Router выбирает акк** по доступности/лимитам (5-часовые окна) + предпочтению юзера.
  Это ровно router/entitlement-шов из ARCHITECTURE.md — валидируем его СЕЙЧАС, без
  self-hosted моделей. Kernel перестаёт быть dormant.
- UI: смена акка + видимость какая сессия на каком акке; выбор модели при запуске (§2).
- Тех-детали (хранение профилей, детект лимитов) — твоя проработка; surface если SDK/CLI
  упирается.

## 6. Харнесс переезжает в brainer (координация с оракулом)

Решение user: агентский харнесс (хуки git-gate/scope-resolve/scope-identity, claude-scope,
шаблоны ролей, доктрина) = **дистрибутив brainer'а** (`knowledger/standards/agents/` уезжает
сюда; knowledger остаётся чистой KB; devopser = чистая инфра). Репо вендорят копию; позже
«установи/синхронизируй харнесс в репо X» = команда продукта.

Это **совместная работа с оракул-архитектором** (перенос из knowledger — его сторона).
Твоя сторона: место в репо (условно `harness/`), стыковка с пресетами (§2 — шаблоны ролей
и хуки-интерпретаторы живут тут). Не начинай перенос в одиночку — синк через user'а.

## 7. Порядок (очередь, WIP=1)

1. **Control-канал** (headless-спавн + send/stream в orchestrator/backend) — §1.
2. **Chater-минимум** (Go: комнаты/сообщения/история + participant-API) — §3.
3. **Мост** participant ↔ session (brainer-сторона).
4. **Tasker-фасад** (контракт + GitHub-адаптер, пересадка Task-board экрана) — §4.

Пресеты (§2) и мульти-акк (§5) — параллелизуемы с очередью по твоему усмотрению (они
про запуск/идентичность, не про чат); харнесс (§6) — по синку с оракулом.

## Вне скоупа (не строить)

`self-hosted`/`peer` провайдеры и agent-loop; биллинг/entitlement-гейт (router = выбор акка,
без тарифов); auth; телеграм-паритет chater (вложения/реакции/поиск/треды); своя начинка
tasker (v0 = прокси); перенос knowledger→brainer в одиночку (§6).
