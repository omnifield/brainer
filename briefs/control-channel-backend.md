## Цель

Backend пересаживается с терминального спавна на **headless-сессии с двусторонним
каналом**: спавн через Claude Agent SDK, send в живую сессию, стрим событий наружу,
resume после рестарта. Терминал перестаёт быть местом жизни агента.

## Что это amends в `backend-mvp.md`

- §Session control: спавн `claude-scope.ps1` как child-процесс **замещается**
  headless-спавном через адаптер. `claude-scope.ps1` — ручной fallback юзера и
  референс env, из репо НЕ выпиливается (догфуд-пересадка §0 direction-брифа —
  только после шагов 1–3 очереди).
- §Телеметрия: activity/статус сессии пересаживаются **на стрим канала**; Loki-путь
  для activity уходит. OTEL/Prometheus остаются строго метрикам (токены/кост).
- «Инъекция сообщений в живую сессию — НЕ MVP» — отменено direction-брифом §1.

## Deliverables

1. **Адаптер claude-code** — реализация kernel-контракта поверх Claude Agent SDK.
   - Живёт ТОЛЬКО здесь (З1 ревью): SDK-типы и семантика границу адаптера не
     пересекают, наружу — только события/операции контракта.
   - Маппинг SDK → события контракта: таблица blueprint §1.4 as-is (включая
     limit-детект по ошибке и `done` из результата хода).
   - `provider_state` handle'а: id сессии SDK + рабочая директория + конфиг-директория
     (все три обязательны — без них resume не находит транскрипт).
   - Permission-маппинг из уровня роли: `readonly → plan`, `standard → acceptEdits +
     allow/deny-списки инструментов`, `trusted → bypassPermissions`. `standard` —
     массовый режим owner'ов: основную нагрузку несут списки инструментов, не режим
     (замечание оракула). `permission-request` в MVP не эмиттится.
   - env при спавне: `OMNIFIELD_SCOPE` + OTEL-инъекция (как в backend-mvp, референс —
     claude-scope).
   - stop: мягкий interrupt по умолчанию, жёсткий kill по флагу.
2. **Resume-on-start** (В2): при старте backend поднять живые handle'ы из
   kernel-реестра; невоскресимую сессию — пометить и отдать событием `error`,
   не падать.
3. **Транспорт наружу** — blueprint §1.5: SSE-стрим событий сессии (reconnect по
   `seq` через Last-Event-ID, события as-is без перевода) + ручка отправки сообщения
   в сессию. Ручки launch/stop расширяются параметрами контракта (role, model).
   Контракт ручек делят фронт+бэк — **менять только через architect**.
4. **Доки** — README: схема потоков (SDK → адаптер → SSE), что осталось у OTEL.

## Вне скоупа

Мост в chater (шаг 3 очереди — подпишется на тот же стрим); мульти-акк
(конфиг-директория уже в `provider_state` — этого достаточно); форвардинг
permission-запросов (горизонт В1-(b)); пресеты (параллельный трек — роль в launch
пока из существующего конфига репо/зон).

## Verify (DoD)

- Реальный e2e: launch headless-сессии → в SSE текут `status`/`message`/`tool-call` →
  send follow-up в живую сессию → агент отвечает в том же контексте → stop.
- Resume-тест: запустить сессию → рестарт backend-процесса → сессия в списке,
  send работает, контекст сохранён.
- Юнит: маппинг SDK→события (SDK замокать), permission-маппинг по ролям,
  limit/error-ветки; SSE reconnect отдаёт пропущенное по `seq` без дублей.
- Тесты + lint зелёные; трейсы на горячих путях (spawn, стрим, resume); доки.

## Заметки

- Git: commit-only, `feat(backend): ...`; push/merge — architect после ревью.
- SDK отдаёт что-то, что не ложится в словарь контракта → STOP + architect
  (расширение словаря — его решение), не изобретай событие и не теряй данные молча.
- Гонки стрима (события до подписки фронта) — буферизуй до первого подписчика или
  отдавай историю по `seq`: механизм на твой выбор, требование — фронт после connect
  видит полную последовательность.
