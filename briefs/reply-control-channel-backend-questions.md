## 2. Статус сессии в списке — in-memory проекция (рекомендация owner ратифицирована)

- `GET /sessions` отдаёт проекцию: последний `state` из `status`/`done` + liveness
  handle'а. Держится в памяти backend (fleet/реестр-слой).
- sqlite-реестр kernel персистит только handle — статус НЕ персистится.
- После рестарта статус деривится из исхода resume-on-start: воскресла → её первый
  `status`; невоскресима → пометка + событие `error` (В2, как в брифе).

## 3. Масштаб amend — переписывание session/provider-слоя, НЕ аддитив

- Терминальный `claude_code.py` — заменить SDK-адаптером под kernel-контрактом.

## 4. seq через рестарт — события НЕ персистить, seq продолжить монотонно

Одна поправка к плану owner:

- Replay-гарантия (reconnect по `Last-Event-ID` с отдачей пропущенного) — в пределах
  жизни процесса, ring-буфер: ок, как планировал.
- НО счётчик `seq` персистится в sqlite-реестре (одно число на сессию) и после
  resume продолжается монотонно. Иначе клиент со старым `Last-Event-ID` ловит
  коллизию дедупа, а мост в chater — сломанный порядок. Персистим курсор, не
  события — «delivery-only» не нарушен.
- В доках зафиксировать границу: reconnect-replay — intra-process; монотонность
  `seq` — на всю жизнь сессии.

реального e2e-прогона (трейс/транскрипт: launch → SSE → send → resume → stop).
